## Problem Statement

大多数用户写提示词时越写越长，试图用详细描述覆盖所有细节。但冗长的描述是在"教"AI，而不是"激活"AI 已有的知识网络。用户需要一个工具，把冗长的描述型提示词转化为精准的概念激活型提示词。

核心洞察：一个精准的人名/理论/框架，能比一万字的描述激活更丰富的 AI 知识。原理来自 Michael Polanyi 的"隐性知识"理论和 Ilya Sutskever 关于模型参数空间的观点。

## What Makes This Cool

"卧槽好牛逼好高大上"是核心用户体验。用户输入一坨模糊想法，skill 炼出一个塔勒布反脆弱框架级别的精准概念，附带一句可以直接复制使用的提示词。

三重惊喜链：
1. 看到自己的烂提示词被炼成金（对比冲击）
2. 发现了自己不知道的概念（认知惊喜）
3. 直接拿到能用的提示词（实用价值）

## Constraints

1. 目标受众：两条线并行 - GitHub 技术用户 + 公众号/小红书 AI 用户
2. 安装体验：一句话命令安装，小白无脑使用
3. 全中文交互
4. 不限定领域（写作、商业、编程、分析等都适用）
5. 风格：不说教、不堆术语，像一个懂提示词的朋友在帮你优化
6. 兼容 Claude Code、Codex、OpenClaw 等 agent 平台
7. 核心目标：开源涨粉，不是商业产品

## Premises

1. **精准概念 > 长描述**：激活 AI 已有知识网络比教 AI 新知识更有效。用户认同：是。
2. **两条线并行**：GitHub 技术版 + 公众号/小红书内容版。skill 是技术实现，内容才是涨粉主力。用户认同：是。
3. **领域通用**：skill 不内置领域逻辑，靠 AI 模型提供领域知识。分析框架是通用的。用户认同：是。
4. **示例优先 + 可选解读**：主流程给可直接复制的示例，可选"解读"深挖概念。快速路径 10 秒，深度路径按需。用户认同：是。
5. **方案 B + 预留知识库接口**：v1 用混合流程，未来可扩展内置知识库。用户认同：是。

## Approaches Considered

### Approach A: 三步线性流程（原始方案）
单 Agent 完成诊断 → 找激活钥匙 → 对比输出。实现简单（S），但结果单一，缺少多角度惊喜。

### Approach B: 混合流程 + 可选解读（推荐方案）
单 Agent 诊断 → 多 Agent 并行推荐不同类型激活钥匙（人名/理论/框架各一个 Agent）→ 单 Agent 对比 → 可选"解读"深挖。
复杂度 M，体验最佳。多角度推荐产生"多专家碰撞"的惊喜感。

### Approach C: 带"知识图谱"的混合流程（未来扩展）
方案 B 全部 + 内置 50-100 个高质量激活钥匙知识库（JSON 文件）。质量最高但工作量大。
v1 预留接口，v2 实现。

## Recommended Approach

**方案 B：混合流程 + 可选解读**

理由：平衡实现成本和用户体验。多 Agent 并行推荐让结果更丰富，可选解读满足不同用户深度需求。预留知识库接口为未来升级铺路。

## 工作流详细设计

### 触发方式
- 命令：`/提示词炼金术` 或 `/prompt-alchemy`（两个都支持）
- description 字段（SKILL.md frontmatter）：`提示词炼金术：把冗长提示词炼成精准概念激活钥匙。触发方式：/提示词炼金术、/prompt-alchemy、「提示词炼金术」「炼金」「提示词优化」`
- 关键词触发：用户消息中包含"炼金"、"提示词优化"、"提示词炼金术"时自动匹配

### 第一步：诊断（单 Agent）

用户输入一段长提示词或描述一个任务目标。

**输入校验规则：**
- 输入少于 20 字 → 提示用户"内容太短了，请补充更多细节或粘贴你的完整提示词"
- 输入模糊度 <= 3（已经足够精炼）→ 告诉用户"你的提示词已经很精炼了"并展示当前提示词的优点，跳过炼金步骤
- 正常输入（模糊度 4-10）→ 进入诊断流程

**模糊度评分锚点：**
- 模糊度 1-3：已使用精准概念（专有名词、框架名称、具体人名）
- 模糊度 4-6：有方向但用描述代替概念（"分析一下市场"而不是"用波特五力模型"）
- 模糊度 7-10：完全泛化，没有具体方向（"帮我写点东西"、"随便分析一下"）

Skill 分析：
- **模糊度评分**：1-10，10 最模糊
- **具体问题**：
  - 是否在用大量描述代替精准概念？
  - 哪几句话可以用一个概念/人名/框架替代？
  - 当前提示词最核心的需求是什么（写作、分析、创意、编程...）？
- **一句话总结**：用户的真实意图

输出格式：
```
🔍 诊断结果

模糊度：7/10
核心意图：{一句话总结}
主要问题：
- {问题1}
- {问题2}
- {问题3}
```

### 第二步：炼金（多 Agent 并行）

**必须使用 Agent tool**：在一条消息中发起 3 个 Agent tool 调用以实现并行。每个 Agent 独立运行，不共享上下文。在不支持并行 Agent 的平台上（如 Claude.ai），降级为串行模式：同一个 Agent 连续运行三次，每次用不同的专家 prompt，每次只看到用户输入和诊断结果（不看到其他专家的输出）。

**Agent 1：人名专家**
- 找出最适合当前任务的思想家/学者/行业人物
- 输出：人名 + 核心思想一句话 + 使用示例

**Agent 2：理论/模型专家**
- 找出最适合当前任务的理论框架或分析模型
- 输出：理论名 + 核心逻辑一句话 + 使用示例

**Agent 3：概念/隐喻专家**
- 找出最适合当前任务的类比、隐喻或方法论
- 输出：概念名 + 核心含义一句话 + 使用示例

v1 版本的 Agent prompt 模板（不包含知识库）：
```
你是提示词炼金术的{类型}专家。

任务：为以下提示词找到最适合的{类型}激活钥匙。

用户原始提示词：
{用户输入}

诊断结果：
{第一步的诊断输出}

推荐规则：
- 必须推荐 2-3 个候选
- 每个候选必须包含：名称、一句话核心说明、可直接复制使用的提示词示例
- 推荐理由要具体，不能泛泛而谈
- 不要推荐太冷门以至于用户完全没听过的（除非它真的特别精准）
- 输出使用中文

诚实规则：
- 如果不确定，承认不确定
- 不强行凑数，宁可推荐 1 个精准的，也不推荐 3 个凑数的
```

输出格式：
```
✨ 炼金结果

👤 人物激活钥匙：
  {人名} — {一句话说明}
  💡 为什么适合：{具体理由}
  📋 直接用：{一句话提示词示例}

📊 理论激活钥匙：
  {理论名} — {一句话说明}
  💡 为什么适合：{具体理由}
  📋 直接用：{一句话提示词示例}

🔑 概念激活钥匙：
  {概念名} — {一句话说明}
  💡 为什么适合：{具体理由}
  📋 直接用：{一句话提示词示例}
```

### 第三步：对比输出（单 Agent）

**"最佳激活钥匙"选择逻辑：** 从三个 Agent 返回的 6-9 个候选中，按以下规则排序选 Top 1 作为"炼金版"：
1. 优先选择诊断步骤中问题最严重的类别对应的钥匙（如用户主要问题是"缺少分析框架"，优先选理论类钥匙）
2. 其次选择与用户核心意图最匹配的（编程任务优先选框架/方法论，写作任务优先选人名/文风）
3. 同等匹配度时，优先选择知名度更高、用户更可能认识的

把用户的原版提示词和炼金后的版本放在一起对比：

```
⚖️ 对比

原版（{字数}字）：
  {用户原始提示词的摘要}

炼金版（{字数}字）：
  {用最佳激活钥匙重写后的简短提示词}

为什么炼金版更有效：
  {1-2句话说明}
```

### 可选第四步：解读（按需，交互式循环）

用户如果对某个激活钥匙感兴趣，可以输入"解读 {钥匙名称}"，skill 会深入解释：
- 这个概念/人/理论是什么
- 核心思想（2-3 个要点）
- 在提示词中怎么用更高级的玩法
- 相关的激活钥匙（扩展推荐）

**这是一个交互式循环**：用户可以连续问多个"解读"，直到用户说"完成"、"不需要了"、"结束"或发送新的提示词输入（重新从第一步开始）。

**新输入 vs 解读延伸的判定规则：**
- 输入包含"解读"+"具体名称" → 解读模式
- 输入超过 50 字且不是对当前概念的追问 → 视为新提示词，重新从第一步开始
- 输入是对当前概念的进一步提问（如"还有类似的理论吗"）→ 继续解读模式

### 未来扩展（v2 知识库）

第二步的 Agent prompt 预留了知识库字段供 v2 扩展，届时读取 JSON 知识库文件为每个 Agent 提供精选的高质量激活钥匙候选列表。v1 完全依赖 AI 模型推荐。

## Open Questions

1. 技能名：`/提示词炼金术` 还是 `/prompt-alchemy`？中文命令对国内用户更友好，但对 GitHub 国际用户不够直观。可以考虑两个都支持。
2. 是否需要支持用户指定"我想要人名类型的激活钥匙"这种筛选？还是每次都三种都推荐？
3. Agent 并行调用的超时处理：如果某个 Agent 超时，是显示部分结果还是重试？

## Success Criteria

1. 用户粘贴一段 200+ 字的提示词，15 秒内看到诊断结果，30-60 秒内完成完整三步流程
2. 炼金结果包含至少 3 个不同类型的激活钥匙，每个都有可直接复制使用的示例
3. 炼金后的提示词比原版短 50% 以上，且效果更好（用户主观判断）
4. GitHub 上有用户 star 和 fork
5. 公众号/小红书相关文章带来可观的阅读量和关注增长

## Distribution Plan

1. GitHub 仓库：开源 skill 代码 + README（中文）+ 安装说明
2. 公众号文章："一条推文火了：一个人名顶十万字提示词" → 引出 skill → GitHub 链接
3. 小红书：短视频/图文展示 skill 的"炼金"效果，附 GitHub 链接
4. README 中包含公众号/小红书二维码，形成双向引流

## What I noticed about how you think

- 你说"肯定要通用的啊"，语气很确定。这不是犹豫后的决定，是你一开始就想清楚了。说明你对这个 skill 的定位很清晰，不需要别人来缩小你的范围。
- 你提了"用户可以进一步深究"这个可选解读的设计。这个细节说明你在想用户体验的分层，不只是"做一个功能"，而是"不同用户用不同方式"。
- 你说"先走 B，留个 C 的接口"。这是典型的 MVP 思维，先交付再迭代。对于一个两个月的学习者来说，这种工程判断力不常见。
